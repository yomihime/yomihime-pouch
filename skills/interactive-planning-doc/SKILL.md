# Interactive Planning Doc

把持续变化的产品或工程讨论整理成可执行的计划文档。目标读者是未来负责实现的 agent，而不是当前聊天参与者。

## 基本原则

- 以用户最新消息为当前事实来源；旧决定与新决定冲突时，更新文档并删掉旧说法。
- 用户说“只写计划”“不改代码”“docs only”“no code changes”时，不编辑实现文件。
- 优先维护一个主计划文档，避免把决策散落到多个文件。
- 先读必要上下文，再写代码库事实；当前源码和稳定架构文档优先于 roadmap、旧计划和聊天记忆。
- 只问会阻塞安全规划的问题；低风险处做合理假设，并在文档里标成假设。
- 如果计划依赖 URL、外部 schema、标准、版本、发布规则或上游状态，先核验来源再写入精确结论。

## 首次响应

触发后先用一两句话说明本轮处理方式：

- 是创建、修订、复查，还是定稿计划。
- 是否会避免代码改动。
- 将读取哪些上下文，或将先创建哪个计划文档。

不要先输出长篇方法论。用户要求直接改文档时，先改文档，再简短汇报。

## 工作流程

### 1. 建立规划契约

确认或推断：

- 规划范围和非目标。
- 是否禁止代码改动。
- 目标文档路径。
- 规划跨度：单功能、重构、版本发布，还是多阶段路线图。
- 文档形态：里程碑、步骤清单，或二者都用。

用户要求“先写空框架”时，先创建框架，只保留简洁占位。

### 2. 收集必要上下文

读取能防止写错的最小上下文：

- `AGENTS.md` 等项目级指令。
- 相关架构文档、路线图和已有计划。
- 能确认模块名、数据路径、当前行为、迁移风险的源码。
- 影响计划的外部格式、API、版本、发布规则或上游状态。

发现文档与源码冲突时，在计划中写明冲突和采用依据。

### 3. 维护决策账本

把用户确认的新决定写入稳定章节，例如 `已确认决策` 或 `Confirmed Decisions`。每条决策写清：

- 决定了什么。
- 建立了什么边界或不变量。
- 是否有后续条件，例如“实现前重新检查上游 PR 状态”。

删除无执行价值的聊天历史、已放弃方案和 agent 自言自语。只有它们解释剩余风险时才保留。

### 4. 面向执行组织文档

按需要包含这些章节；不服务于未来执行的章节应删除：

- 目的与范围。
- 非目标。
- 已确认决策。
- 当前状态与目标状态。
- 数据模型、路径、持久化和迁移说明。
- UI 与用户工作流。
- 模块边界和代码结构。
- 外部格式或集成映射。
- 错误处理、隐私、安全和用户数据规则。
- 细粒度里程碑。
- 验证与代码自审查要求。
- 提交分组。
- 用户验收流程。
- 验收后收尾、合并和版本准备。
- 待确认问题。

### 5. 写具体里程碑

高风险计划拆成多个小里程碑。每个里程碑要窄到执行者能判断是否完成。

每个里程碑优先写：

- `交付`：可观察结果。
- `验收`：如何检查。
- `边界`：不得修改或不得耦合的内容。

避免“优化架构”“完善 UI”这类笼统项。拆成数据模型、存储、页面骨架、交互、worker、预览、导入导出、验证、文档和兼容性等独立里程碑。

用户只要求里程碑时，不把计划改写成逐步操作清单；里程碑仍然可以很多、很细。

### 6. 单独设计提交节奏

不要默认最后一个大提交，也不要机械地一个里程碑一个提交。按可审查阶段分组：

- 页面或职责边界调整。
- 数据模型与存储边界。
- UI 骨架与导航。
- 编辑、资产和媒体工作流。
- 编译、渲染和预览核心。
- 导入导出与外部格式。
- 文档与 TODO 同步。
- 版本和发布准备。

每个提交分组写明覆盖里程碑、建议提交信息和提交前检查。提交信息遵循项目约定。

### 7. 区分开发与验收后收尾

开发里程碑属于开发计划；描述已实现行为的文档更新也属于开发工作。

验收后收尾只在以下条件满足后开始：

- 所有计划里程碑已完成。
- 用户已经试用检查。
- 已知 bug 已修复。
- 必要 UI 布局调整已完成。

收尾可包含删除临时桥接代码、调试入口、feature flag、手工测试产物、日志、缓存、重复兼容路径、中间文件，以及版本和发布准备。若收尾时发现行为 bug，回到对应开发里程碑或验收修正阶段。

### 8. 规划架构边界

计划涉及代码结构时，提前命名边界：

- 核心模型和领域服务。
- 存储与路径 helper。
- 导入、导出和渲染器。
- UI 页面、组件、弹窗和 worker。
- 页面之间的信号或事件。
- 迁移与旧数据兼容面。
- 依赖不得泄漏的位置。

保持低耦合：UI 触发工作，worker 执行耗时任务，core 承担业务逻辑，storage helper 负责文件系统路径，renderer 负责派生产物格式。

### 9. 加入自审查约束

大型计划要有代码自审查章节。要求执行者检查：

- 架构边界。
- 死代码和重复逻辑。
- 命名与数据所有权。
- 用户数据安全。
- 错误文案和 i18n。
- 外部格式兼容。
- 测试或验证命令。
- 临时开发桥接是否已删除或有保留理由。

每个提交分组提交前做局部自审查；进入用户验收前再做一次整体自审查。

### 10. 干净迭代

用户改变决定时：

- 更新文档里的权威章节。
- 删除与新决定冲突的旧文本。
- 检查相关里程碑、提交分组、收尾和验证章节。
- 简短说明本轮改了什么。

用户要求提交、amend 或 push 时，先确认工作区只包含预期的计划文件变更。

### 11. 定稿前模拟执行者

接近定稿时，站在“下一位执行 agent 没参与讨论”的角度检查：

- 是否知道先读哪些文件。
- 是否知道哪些行为不能改。
- 是否知道每个里程碑完成后怎么验收。
- 是否知道哪些内容等用户确认，哪些可以自主执行。
- 是否知道提交如何分组。
- 是否知道验收后收尾何时开始，且不会把 bug 修复塞进收尾。

发现含糊点时，直接改文档或向用户提出最少量问题。

## 最小模板

```markdown
# <计划标题>

> 本文档是执行计划，不代表实现状态。

## 1. 范围

## 2. 非目标

## 3. 已确认决策

## 4. 当前状态与目标状态

## 5. 数据与持久化

## 6. UI 与工作流

## 7. 代码结构

## 8. 里程碑

### M01：<具体里程碑>

交付：

- ...

验收：

- ...

## 9. 验证与自审查

## 10. 提交分组

## 11. 验收后收尾

## 12. 待确认问题
```

## 质量检查

定稿前确认：

- `description` 仍能准确触发本 skill，且没有混入只属于单个项目的细节。
- 未参与讨论的 agent 也能按计划执行。
- 已确认决策容易找到。
- 剩余问题少且明确。
- 里程碑足够细，不容易被宽泛理解。
- 提交分组少于里程碑，并对应可审查阶段。
- 收尾不是未完成开发工作的垃圾桶。
- 相关时包含版本、发布、迁移和用户数据规则。
- 没有把可从 `AGENTS.md`、架构文档或源码读取的信息重复写成长期噪音。
- 文档没有多余赞美、闲聊、历史争论或已废弃方案。
